# wg-split 2.0.0 — дизайн-документ

Дата: 2026-06-24
Ветка: `feature/2.0.0`
Статус: реализовано (2026-06-26) — все 5 столпов + транспорт-агностичный шов; off-box
проверки (shellcheck/bats/selftest) зелёные; ждёт on-box проверки на стенде (§5).

## 1. Контекст и текущее состояние

`wg-split` (1.7.3) — локальный split-tunnel шлюз для OpenWrt: LAN-трафик
policy-маршрутизируется по приоритетным WireGuard/AmneziaWG туннелям с failover,
списки IP/доменов рулят что идёт в VPN, опциональный zapret DPI-bypass на WAN.

Состав:
- shell-ядро `wg-split/files/usr/local/{sbin,lib}` — failover-демон, doctor,
  apply, загрузчики списков, common.sh;
- LuCI-приложение `luci-app-wg-split` — одностраничная панель статуса + форма;
- UCI-конфиг `/etc/config/wg-split`; русская вики `docs/ru`.

Развёрнуто и проверено на живом стенде:
- **Сайт A** `10.8.0.2` (OpenWrt 25.12.4, mediatek/filogic): LAN `10.8.1.0/24`,
  туннель AmneziaWG `awg0=10.8.0.2/24`, zapret активен, doctor `overall OK`.
- **Сайт B** `10.8.0.3`: LAN `10.8.2.0/24`, туннель `wg0=10.8.0.3/24`.
- Оба — спицы к одному хабу-VPS `45.144.53.1:500` (hub-and-spoke).

### Узкие места 1.7.3 (обоснование 2.0.0)
- **LuCI ↔ backend через `fs.exec`** прямого вызова скриптов → требует широкого
  ACL `file:exec`, не least-privilege, не каноничный путь LuCI.
- **Нет структурного журнала failover** — только syslog, события не агрегированы.
- **Нет учёта трафика** по туннелям (хотя `wgshow transfer` под рукой).
- **Статус и настройки на одной странице** — нет разделения dashboard/settings.
- **Главная боль пользователя — firewall**: туннель не в зоне / нет masq / нет
  forwarding → три `FAIL` «решай сам», потому что пакет принципиально не трогает
  firewall.
- **Site-to-site не first-class**: приватные подсети соседних спиц не
  маршрутизируются в туннель без ручного `vpn_cidr` (см. §4.5).
- **Нет quality-gate**: только `WG_SPLIT_SELFTEST` внутри doctor, нет CI-линта/тестов.

## 2. Принципы

### 2.1 Обратная совместимость конфига
UCI-схема меняется **только аддитивно** (без переименований/удалений). Конфиги
1.7.3 продолжают работать as-is. Миграция (`/etc/uci-defaults`) лишь идемпотентно
добавляет новые дефолты. «Major» оправдан новой ubus-поверхностью, реструктуризацией
UI и firewall/site-to-site функциями, а не ломкой конфига.

### 2.2 Транспорт-агностичность (задел под 3.0.0 = vless/hysteria через sing-box)
3.0.0 добавит outbound-протоколы (VLESS/Hysteria) поверх sing-box как
альтернативу WG/AWG. Чтобы это не потребовало переписывания ядра, **2.0.0
вводит абстракцию endpoint с полем `type`**:

- `config endpoint` получает опцию `type` (по умолчанию `wg`, что покрывает
  wireguard и amneziawg — текущее поведение). В 2.0.0 поддерживается только `wg`.
- Вся type-зависимая логика проходит через узкие dispatch-хелперы в `common.sh`:
  - `ep_present <iface>` — есть ли живое egress-устройство;
  - `ep_liveness <iface>` — источник «живости» (для `wg` = handshake-age; для
    sing-box в 3.0 — статус процесса/health-порта);
  - `ep_egress_dev <iface>` — L3-устройство для `ip route … dev`;
  - `ep_bringup` / `ep_restart` — поднять/перезапустить (для `wg` = ifup/ifdown).
- Failover, doctor, ubus и UI оперируют **endpoint**, а не «WG iface». UI не
  хардкодит «WireGuard»/«handshake» — обобщённые ярлыки «endpoint»/«liveness»/
  «last seen».

2.0.0 **не реализует** sing-box — только оставляет швы. `type=singbox` и менеджер
sing-box — предмет 3.0.0.

## 3. Архитектура верхнего уровня

```
            ┌──────────────── LuCI (htdocs/.../view/wg-split) ───────────────┐
            │  status.js (Dashboard)        settings.js (Settings form)      │
            └───────────────┬───────────────────────────┬────────────────────┘
                            │ rpc.declare (ubus)         │ form.Map (uci)
                            ▼                            ▼
                 ┌─────────────────────┐         /etc/config/wg-split
                 │ rpcd: object wg-split│
                 │  status/events/action│
                 └──────────┬───────────┘
                            │ exec (thin wrappers)
       ┌────────────────────┼─────────────────────────────────────────┐
       ▼                    ▼                    ▼                      ▼
  wg-split-doctor     wg-split-failover    wg-split-firewall      update-* / apply
  (--json/--events)   (+ journal())        (check/fix)            (без изменений ядра)
                            │
                            ▼  пишет события
                  /var/run/wg-split-events (кольцевой буфер)
```

## 4. Столпы

### 4.1 Pillar 1 — Dashboard 2.0
**Цель:** превратить страницу в операторский дашборд.

- **Разделение меню** `admin/services/wg-split` на два view + подпункты:
  - `…/wg-split` (lander) → `status.js`: hero + цепочка + таблицы + диагностика +
    тулбар + **живой трафик** + **таймлайн**.
  - `…/wg-split/settings` → `settings.js`: только `form.Map`.
  - `menu.d` получает второй пункт; общий ACL.
- **Живой трафик rx/tx**: doctor добавляет в `endpoints[]` поля `rx`/`tx` (байты
  из `wgshow <if> transfer`; для не-wg типов — из `ep_*` диспетчера, в 2.0 пусто).
  UI считает скорость между поллами и показывает на узлах цепочки и в таблице.
- **Таймлайн failover**: §4.3.

**Файлы:** `view/wg-split/status.js` (новый, выносится из текущего settings.js),
`view/wg-split/settings.js` (ужимается), `menu.d/luci-app-wg-split.json`,
`wg-split-doctor` (поля трафика).

### 4.2 Pillar 2 — Firewall auto-fix (opt-in)
**Цель:** в один клик закрыть три firewall-FAIL.

Модель зоны **точно повторяет рабочую зону `awg` с 10.8.0.2** (подтверждено
живым `uci show firewall`):
```
config zone
    option name '<iface>'
    option input 'ACCEPT'
    option output 'ACCEPT'
    option forward 'ACCEPT'
    option masq '1'
    option mtu_fix '1'
    list network '<iface>'
config forwarding   # lan -> zone
config forwarding   # zone -> lan
config forwarding   # zone -> wan
```
(accept-all + masq, без `forward REJECT`).

- Новый скрипт **`wg-split-firewall`**:
  - `check <iface>` — чего не хватает (переиспользует `fw_*` из common.sh).
  - `fix <iface>` — идемпотентно: если iface уже в зоне — добиваем masq/forwarding;
    иначе создаём зону по модели выше + forwarding `lan-zone↔зона` и `зона→wan`;
    `uci commit firewall && fw4 reload`. Имя LAN-зоны берётся из `fw_lan_zone`.
- **UI**: на каждом firewall-`FAIL` рядом со ссылкой «Firewall settings» —
  кнопка **«Починить автоматически»** → `action {name:fw_fix, iface}` → refresh.
  Действие явное (кнопка + confirm), без тихого автоприменения.
- **Визард первого запуска**: текущий `firstRun`-блок → шаги «Создать туннель из
  обнаруженного awg/wg» → «Настроить firewall» (fw_fix) → «Проверить».

**Файлы:** `wg-split/files/usr/local/sbin/wg-split-firewall` (новый),
`status.js` (кнопки/визард), ACL + ubus action.

### 4.3 Pillar 3 — Журнал событий + rpcd/ubus
**Журнал (backend под таймлайн):**
- `common.sh`: хелпер `journal <kind> <from> <to> <reason>` → дописывает строку
  `ts \t kind \t from \t to \t reason` в кольцевой буфер `/var/run/wg-split-events`
  (RAM, без износа флеша, ~200 строк, обрезается). Виды: `switch`, `recover`,
  `restart`, `killswitch`, `zapret_fallback`, `wan_fallback`.
- `wg-split-failover`: вызвать `journal` в существующих точках (`active tunnel ->`,
  `all tunnels down ->`, `restarting stuck iface`).
- `wg-split-doctor --events` → JSON-массив последних N событий.

**rpcd/ubus:**
- Плагин **`/usr/libexec/rpcd/wg-split`** (shell, в стиле проекта), объект
  `wg-split` с методами:
  - `status` → `wg-split-doctor --json`;
  - `events` → `wg-split-doctor --events`;
  - `action {name, iface?}` где name ∈ `apply|restart|disable|update_ipsum|
    update_ru|update_domains|fw_fix` → запускает соответствующий скрипт, возвращает
    `{code, stdout}`.
- **ACL** `luci-app-wg-split.json`: вместо широкого `file:exec` — пометодный
  `ubus` доступ к `wg-split` (`status/events/action`). `read` = status/events,
  `write` = action. Широкие `file:exec` убираются.
- LuCI-views зовут `rpc.declare`. Shell-скрипты остаются для SSH (ubus-методы —
  тонкие обёртки → низкий риск, легко откатить).

**Файлы:** `common.sh` (`journal`), `wg-split-failover` (вызовы), `wg-split-doctor`
(`--events`), `usr/libexec/rpcd/wg-split` (новый), `acl.d/luci-app-wg-split.json`,
`status.js` (rpc вместо fs.exec).

### 4.4 Pillar 4 — Release-гигиена
- **CI** (`.github/workflows`): job `lint` (shellcheck по `sbin/*`,`lib/*`) +
  job `test` (`WG_SPLIT_SELFTEST=1` и новые **bats** на чистую логику:
  `clean_ip_list`, `fw_bool`, `json_esc`, `_rank`, парсинг endpoints).
- **Миграция** `/etc/uci-defaults/99-wg-split-migrate` — аддитивно (напр. дефолт
  `type=wg` на существующие endpoint-секции, новые опции).
- **Device picker**: поле `ip` в `device` → datalist из `network.getHostHints()`
  (IP/MAC/hostname из DHCP).
- **Docs**: `docs/en/` паритет с `docs/ru`, обновление README, `CHANGELOG.md` для
  2.0.0, roadmap → отмечен сделанным.
- **Баг-чек**: `ru count=0` при `ru_enabled=1` (замечен на стенде) — разобрать,
  это корректно (в blocklist ru-сет не подключён) или подвести в doctor.

### 4.5 Pillar 5 — Site-to-site (межсайтовые подсети)
**Контекст:** пользователь наступил на кейс — LAN соседней спицы (`10.8.2.0/24`)
была недоступна с клиентов за `10.8.0.2`, потому что в blocklist приватные подсети
не помечаются в туннель. Хаб уже делегирует подсети спицам; не хватало только
маркинга на стороне спицы. **Немедленный фикс уже применён** на обоих роутерах
(`vpn_cidr += соседняя LAN`, `apply`), путь проверён двусторонне
(`ip route get … mark 0x40000 → dev <tun> table 200`).

**Что делает пакет в 2.0.0:**
- В «Manual entries» «VPN subnets» получает явный site-to-site смысл: ярлык/подсказка
  «Удалённые LAN-подсети пиров (site-to-site)».
- **doctor-проверка**: для каждого `vpn_cidr` — если `ip route get <addr> mark
  $WG_MARK` уходит НЕ в активный туннель (а в WAN) → `FIXABLE` «подсеть в VPN-списке
  не маршрутизируется в туннель, запусти wg-split-apply».
- *(Опционально, оценить)* авто-обнаружение соседних спиц в WG-транзитной /24 и
  предложение добавить их LAN-подсети.

**Файлы:** `settings.js` (ярлык/подсказка), `wg-split-doctor` (проверка), docs.

## 5. Стратегия тестирования
- **Off-box**: `WG_SPLIT_SELFTEST=1 wg-split-doctor`, bats на чистую логику, shellcheck — в CI.
- **On-box (живой стенд)**: после каждого столпа — сборка `.apk` (CI/SDK),
  `apk add`, прогон `wg-split-doctor`, проверка ubus (`ubus call wg-split status`),
  клик по LuCI. Самый рискованный столп — rpcd/ubus: проверяем ubus-вызовы до
  удаления `fs.exec`.
- `10.8.0.3` — как пир для сквозной site-to-site проверки.

## 6. Порядок и риск
1. Pillar 5 doctor-проверка + Pillar 4 каркас (CI/bats) — низкий риск, быстрый фундамент.
2. Pillar 1 разделение view + трафик.
3. Pillar 3 журнал + таймлайн (backend → UI).
4. Pillar 2 firewall auto-fix + визард.
5. Pillar 3 rpcd/ubus миграция (последней, самый риск; до неё UI на fs.exec).
6. Pillar 4 docs/migration/CHANGELOG, bump VERSION → 2.0.0.

Транспорт-агностичный шов (§2.2) закладывается в Pillar 1/3 по ходу (поле `type`,
`ep_*` диспетчеры), не отдельным этапом.

## 7. Вне охвата
- Реализация sing-box / vless / hysteria (это 3.0.0 — только оставляем швы).
- IPv6-маршрутизация.
- Управление WG-ключами/пирами.
- Переписывание routing-ядра без диагностической нужды.

## 8. Полевые наблюдения (срез прода 27.06, A=10.8.0.2 / B=10.8.0.3, оба 1.7.3)
Оба роутера стабильны (uptime 4д / 3д20ч, fail_count=0, ноль фейловеров, свежие
handshake'и, AmneziaWG-обфускация активна, s2s-фикс от 26.06 жив в ruleset на обоих).
Из живых данных всплыли 4 дефекта — заводятся в 2.0.0:

- **N1 — лог-флуд `nozapret synced` (→ Pillar 3, при journal'е).** `set_vpn` на каждом
  tick (180с) логирует `nozapret synced: N entries`; в `logread` это 23% строк на A и
  40% на B, volatile-буфер дотягивается лишь до ~Jun 26 → история за 25.06 уже
  вытеснена. Лечение: понизить этот лог до debug / писать только при изменении; реальные
  события уводим в ring-buffer журнал (`journal()`), а не в syslog-флуд.
- **N2 — холостой ре-синк nozapret (→ Pillar 4 / failover).** `set_vpn` безусловно гоняет
  `rebuild_nozapret 1` каждый tick — перезаливает ~71к строк в set при неизменном
  состоянии (CPU + лог). Сделать идемпотентным: пропускать, если входные
  файлы/состояние не менялись (хеш `RU_FILE`+`IPSUM_FILE`+`INCLUDE_VPN`, или mtime-guard).
- **N3 — doctor врёт по `ru` в blocklist (→ Pillar 1/3, doctor).** `ru` КОРРЕКТНО
  вклинивается в nozapret (`wg-split-sync-nozapret:36 cat $RU_FILE`), но doctor читает
  пустой `wg_split_ru_subnets_v4` (он задействован лишь в `full`) и рапортует
  `ru count:0` — диагностика лжёт на рабочем списке. Лечение: в blocklist режиме
  показывать вклад ru через nozapret (источник = строки файла), а не live-set RU_SET;
  `count`/`ok` считать от фактической роли списка в текущем режиме.
- **N4 — несогласованная семантика `count` в doctor (→ Pillar 1/3, doctor).** Поля `count`
  мешают «строки в файле» и «элементы в live-set» (ipsum: файл 27372 / set 10490 / doctor
  20980; nozapret: лог «71345» vs live 46207 из-за auto-merge интервалов). Развести явно:
  `file_count` vs `set_count`, и не пугать расхождением, которое даёт interval-merge.

**FYI (не код, эксплуатация):** на каждом сайте один endpoint (priority 1) — failover'у
не на что переключаться; при падении единственного хаб-туннеля сразу wan/zapret/killswitch.
Мультиэндпоинт пакет уже умеет — вопрос конфигурации стенда, не 2.0.0.
